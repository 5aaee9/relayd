# API

`relayd` exposes an authenticated HTTP control-plane API under `/v1` and an authenticated Prometheus scrape endpoint at `/metrics`.

## Quick start

```bash
export BASE_URL=http://127.0.0.1:8080
export AUTH_TOKEN=mytoken

curl -sS \
  -H "Authorization: Bearer ${AUTH_TOKEN}" \
  "${BASE_URL}/v1/allocations"
```

## Authentication

All endpoints require a bearer token:

```http
Authorization: Bearer <AUTH_TOKEN>
```

If the header is missing, malformed, or the token does not match, the server returns:

- `401 Unauthorized`
- body: `unauthorized`

## Conventions

- Base path: `/v1` for JSON control-plane endpoints; `/metrics` is the Prometheus scrape endpoint without the `/v1` prefix
- Successful `POST`/`PUT`/`GET` responses use `Content-Type: application/json`
- Successful `DELETE` responses return `204 No Content` with an empty body
- Error responses are plain text, not JSON
- Allocation IDs are UUIDv7 strings generated by the server
- `host` accepts IP literals only, for example `127.0.0.1` or `::1`
- `protocol` values are `tcp`, `udp`, and `both`
- `GET /v1/allocations` and `GET /v1/ports` return rows ordered by `protocol`, then `port` (currently lexical protocol order: `both`, `tcp`, `udp`)

## Resource model

### Allocation object

```json
{
  "id": "018f0d5d-86d6-7e57-8bb5-c0b7c38e2f0e",
  "protocol": "tcp",
  "port": 10000,
  "created_at_ms": 1712822400000,
  "updated_at_ms": 1712822400000
}
```

For `protocol = "both"`, the allocation owns both the TCP and UDP listener on the same numeric `port`. It is still represented as one allocation object with one `id`.

### Binding object

```json
{
  "allocation_id": "018f0d5d-86d6-7e57-8bb5-c0b7c38e2f0e",
  "host": "127.0.0.1",
  "target_port": 8080,
  "effective_target_port": 8080,
  "effective_host": "127.0.0.1",
  "runtime_status": "active",
  "error_kind": null,
  "last_error": null,
  "created_at_ms": 1712822401000,
  "updated_at_ms": 1712822401000
}
```

### Compatibility aggregate (`GET /v1/ports`)

This endpoint remains available as an aggregate/compatibility read model. It keeps the legacy flat shape, but `target_port` may now be `null` when an allocation has no binding. A dual-protocol allocation is returned as one row with `protocol = "both"`; clients should not expect separate TCP and UDP rows for the same allocation.

```json
{
  "id": "018f0d5d-86d6-7e57-8bb5-c0b7c38e2f0e",
  "protocol": "tcp",
  "port": 10000,
  "target_port": 8080,
  "host": "127.0.0.1",
  "effective_target_port": 8080,
  "effective_host": "127.0.0.1",
  "host_configured": true,
  "runtime_status": "active",
  "error_kind": null,
  "last_error": null,
  "created_at_ms": 1712822400000,
  "updated_at_ms": 1712822405000
}
```

### `runtime_status` values

For `protocol = "both"`, `active` means both TCP and UDP listeners are active for the shared binding target. If either listener cannot be applied, the aggregate status is degraded and `last_error` identifies the failure where available.

- `active`
- `rejecting_no_host`
- `degraded_bind_failed`
- `degraded_apply_failed`
- `degraded_close_failed`

### `error_kind` values

- `bind_failed`
- `apply_failed`
- `close_failed`
- `restore_failed`
- `invalid_input`

## Primary endpoints

### Allocation endpoints

#### Create allocation

`POST /v1/allocations`

Request body:

```json
{
  "protocol": "tcp"
}
```

Success:

- `201 Created`
- body: allocation object

Notes:

- `protocol` must be `tcp`, `udp`, or `both`
- An allocation can exist without any binding
- The relay port is picked automatically from `PORT_RANGE`
- `both` reserves one numeric relay port for both TCP and UDP listeners
- A `both` allocation conflicts with any existing allocation on the same port; `tcp` and `udp` allocations also conflict with an existing `both` allocation on the same port

#### List allocations

`GET /v1/allocations`

Success:

- `200 OK`
- body: array of allocation objects

#### Get allocation

`GET /v1/allocations/{id}`

Success:

- `200 OK`
- body: allocation object

If the allocation is missing:

- `404 Not Found`

#### Delete allocation

`DELETE /v1/allocations/{id}`

Success:

- `204 No Content`

Semantics:

- Deletes the allocation
- Deletes the binding too when one exists

### Binding endpoints

#### Create or replace binding

`PUT /v1/allocations/{id}/binding`

Request body:

```json
{
  "host": "127.0.0.1",
  "target_port": 8080
}
```

Success:

- `200 OK`
- body: binding object

Semantics:

- Replaces the entire binding for the allocation
- For `protocol = "both"`, the same `host` and `target_port` apply to both TCP and UDP forwarding
- If the allocation does not exist, returns `404`

#### Get binding

`GET /v1/allocations/{id}/binding`

Success:

- `200 OK`
- body: binding object

If the allocation has no binding:

- `404 Not Found`

#### Delete binding

`DELETE /v1/allocations/{id}/binding`

Success:

- `204 No Content`

Semantics:

- Removes the binding only
- Keeps the allocation
- A follow-up `GET /v1/allocations/{id}` still succeeds
- A follow-up `GET /v1/allocations/{id}/binding` returns `404`

## Compatibility endpoints

These endpoints remain available for legacy clients. New clients should prefer the allocation/binding endpoints above.

### Create legacy port record

`POST /v1/ports`

Request body:

```json
{
  "protocol": "tcp",
  "target_port": 8080
}
```

Success:

- `201 Created`
- body: compatibility aggregate object

Semantics:

- Creates an allocation
- `protocol` must be `tcp`, `udp`, or `both`
- `both` has the same shared-port reservation semantics as `POST /v1/allocations`
- Seeds a compatibility binding with `target_port`
- Until `host` is configured, `runtime_status` is typically `rejecting_no_host`

### Set target host (legacy)

`POST /v1/ports/target`

Request body:

```json
{
  "id": "018f0d5d-86d6-7e57-8bb5-c0b7c38e2f0e",
  "host": "127.0.0.1"
}
```

Success:

- `200 OK`
- body: compatibility aggregate object

### Update legacy port record

`POST /v1/ports/{id}`

Request body:

Any non-empty combination of:

```json
{
  "target_port": 9090,
  "host": "127.0.0.1"
}
```

Success:

- `200 OK`
- body: compatibility aggregate object

Validation:

- At least one of `target_port` or `host` must be present
- Empty body `{}` returns `400 Bad Request` with body `empty update`

### Delete legacy port record

`DELETE /v1/ports/{id}`

Success:

- `204 No Content`

Semantics:

- Deletes the allocation and any binding

### List aggregate compatibility view

`GET /v1/ports`

Success:

- `200 OK`
- body: array of compatibility aggregate objects

Dual-protocol allocations appear once with `protocol = "both"`, one `id`, and one `port`.

## Metrics

### Prometheus metrics endpoint

`GET /metrics`

Exports scrape-ready Prometheus text exposition for per-listener live connection/session counts and rx/tx speeds.

Authentication:

- Requires the same `Authorization: Bearer <AUTH_TOKEN>` header as the `/v1` control-plane endpoints.
- Missing, malformed, or wrong bearer tokens return `401 Unauthorized` with body `unauthorized`.

Success:

- `200 OK`
- `Content-Type: text/plain; version=0.0.4; charset=utf-8`
- body: Prometheus text exposition format

Exported metrics:

```text
relayd_connections_current{port="<port>",protocol="tcp|udp"} <current_sessions>
relayd_rx_bytes_per_second{port="<port>",protocol="tcp|udp"} <bytes_per_second>
relayd_tx_bytes_per_second{port="<port>",protocol="tcp|udp"} <bytes_per_second>
```

Labels:

- `port` is the concrete relay listener port.
- `protocol` is the concrete listener protocol, `tcp` or `udp`.
- A `protocol = "both"` allocation is exported as two series for the same port: one with `protocol="tcp"` and one with `protocol="udp"`.
- Allocation IDs are not exported as labels.

Speed semantics:

- `rx` is bytes accepted by relayd from external clients on the allocated listener port.
- `tx` is bytes sent by relayd back to external clients on the allocated listener port.
- Speeds are calculated from per-listener byte-total deltas between scrapes. The first scrape for a new `port`/`protocol` label reports `0` for rx/tx speed because no previous sample exists.
- When an allocation is deleted, its label is omitted from later scrapes.

Example:

```bash
curl -sS \
  -H "Authorization: Bearer ${AUTH_TOKEN}" \
  "${BASE_URL}/metrics"
```

### JSON metrics endpoint

`GET /v1/metrics`

Returns existing runtime counters and gauges as a single JSON object for compatibility. This endpoint remains JSON and is separate from Prometheus `/metrics`.

Success:

- `200 OK`
- `Content-Type: application/json`
- body: JSON object with integer values

## Error mapping

Current HTTP-level behavior:

- `400 Bad Request` — invalid protocol / host / port / empty legacy update
- `401 Unauthorized` — missing or invalid bearer token
- `404 Not Found` — allocation or binding not found
- `409 Conflict` — no available port
- `503 Service Unavailable` — runtime apply/update/delete timeout or failure
- `500 Internal Server Error` — unexpected internal error
